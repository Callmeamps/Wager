Wager Tracker Upgrade — SvelteKit + SQLite/Dolt

Executive Summary

Project: Upgrade Wager/Dare Tracker from MVP (Express + vanilla HTML + JSON file) to production-grade application.
Stack: SvelteKit (frontend+API), SQLite (embedded) or Dolt (versioned), Tailwind CSS, Cloudflare/Netlify/Vercel hosting.
Goal: Reliable, secure, maintainable wager tracking with real-time updates, audit trail, and scalable foundation.

Current State Problems

Vision & Goals

Primary Goals

  1. Reliability: No data loss, consistent state, handle concurrent access.
  2. Security: Authentication, authorization, input validation, HTTPS, security headers.
  3. Maintainability: Clean architecture, tested, documented, modular.
  4. User Experience: Responsive, real-time updates, intuitive flows, error feedback.
  5. Observability: Logging, health checks, metrics (optional phase 2).
  6. Portability: Deploy anywhere (edge-friendly, Docker).

Success Metrics

  • Zero data corruption in concurrent edit tests
  • 100% API route test coverage
  • Lighthouse accessibility + performance >90
  • <200ms average API response time (p95)

Technology Choices

SvelteKitSSR+SSG, file-based routing, built-in API routes, smaller bundles, edge runtime
DoltGit-like versioning, MySQL-compatible SQL, branch/merge/audit built-in
SQLiteZero config, single file, fast for small datasets; fallback if Dolt complexity too high initially
PrismaType-safe ORM, migrations, multi-db support
Tailwind CSSUtility-first, consistent design system
Lucia/Better-AuthSession auth without JWT complexity
Port Allocation: Avoid common ports. Use 5173 (dev) or 4317 (prod) for SvelteKit. Dolt PostgreSQL default is 5432.

Data Model

Prisma Schema

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "sqlite" // or "postgresql" for Dolt
  url      = env("DATABASE_URL")
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String?
  avatar    String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  createdWagers Wager[] @relation("CreatorWagers")
  participations WagerParticipant[]
  votes        Vote[]
  proofs       Proof[]
}

model Wager {
  id           String   @id @default(uuid())
  title        String
  description  String?
  stakeAmount   Decimal?
  stakeCurrency String  @default("USD")
  deadline     DateTime?
  status       WagerStatus @default(PENDING)
  resolution   WagerResolution?
  createdBy    String
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  creator    User    @relation("CreatorWagers", fields: [createdBy], references: [id])
  participants WagerParticipant[]
  proofs      Proof[]
  votes       Vote[]
  comments    Comment[]

  @@index([status])
  @@index([deadline])
  @@index([createdBy])
}

model WagerParticipant {
  wagerId     String
  userId      String
  joinedAt    DateTime @default(now())

  wager Wager @relation(fields: [wagerId], references: [id], onDelete: Cascade)
  user  User  @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@id([wagerId, userId])
}

model Proof {
  id        String   @id @default(uuid())
  wagerId   String
  userId    String
  link      String
  evidence  String?
  createdAt DateTime @default(now())

  wager Wager @relation(fields: [wagerId], references: [id], onDelete: Cascade)
  user  User  @relation(fields: [userId], references: [id])

  @@index([wagerId])
  @@index([userId])
}

model Vote {
  id       String   @id @default(uuid())
  wagerId  String
  voterId  String
  isValid  Boolean
  comment  String?
  createdAt DateTime @default(now())

  wager  Wager @relation(fields: [wagerId], references: [id], onDelete: Cascade)
  voter  User  @relation(fields: [voterId], references: [id])

  @@unique([wagerId, voterId])
  @@index([wagerId])
  @@index([voterId])
}

model Comment {
  id       String   @id @default(uuid())
  wagerId  String
  userId   String
  content  String
  createdAt DateTime @default(now())

  wager Wager @relation(fields: [wagerId], references: [id], onDelete: Cascade)
  user  User  @relation(fields: [userId], references: [id])

  @@index([wagerId])
  @@index([userId])
}

enum WagerStatus {
  PENDING
  SUBMITTED
  VOTING
  RESOLVED
  CANCELLED
}

enum WagerResolution {
  WON
  LOST
  TIE
  DISPUTED

Audit Trail

If using SQLite, create AuditLog model with: action, table, recordId, changedBy, changedAt, oldValue, newValue (JSON).
If using Dolt: built-in history via dolt sql -q "SELECT * FROM dolt_diff..." or dolt log.

API Design

All endpoints under /api/v1/*. Use JSON responses, proper HTTP status codes, consistent error format:

{
  "error": true,
  "message": "Human-readable description",
  "details": {} // optional context
}

Authentication

  • POST /api/v1/auth/register — { email, password, name? }
  • POST /api/v1/auth/login — { email, password } → sets httpOnly cookie
  • POST /api/v1/auth/logout — clears session
  • POST /api/v1/auth/refresh — refresh access token

Wager Endpoints

  • GET /api/v1/wagers — query: ?status=pending&limit=20&offset=0
  • POST /api/v1/wagers — { title, description, stake?, participants[], deadline? }
  • GET /api/v1/wagers/:id
  • PATCH /api/v1/wagers/:id — status change, edit (creator only)
  • DELETE /api/v1/wagers/:id — soft delete (status=cancelled)
  • POST /api/v1/wagers/:id/participants — { userId }
  • DELETE /api/v1/wagers/:id/participants/:userId

Proof Endpoints

  • GET /api/v1/wagers/:id/proofs
  • POST /api/v1/wagers/:id/proofs — { link, evidence? }
  • DELETE /api/v1/proofs/:id

Vote Endpoints

  • GET /api/v1/wagers/:id/votes
  • POST /api/v1/wagers/:id/votes — { isValid: boolean, comment?: string }
  • DELETE /api/v1/votes/:id

Comment Endpoints

  • POST /api/v1/wagers/:id/comments — { content }
  • PATCH /api/v1/comments/:id — { content }
  • DELETE /api/v1/comments/:id

Health & Metrics

  • GET /api/v1/health — { status: 'ok', timestamp, db: 'connected' }
  • GET /api/v1/metrics — Prometheus format (optional)

Key Components

  • WagerCard: status color-coded (yellow pending, blue voting, green resolved, red cancelled). Click → detail page.
  • WagerForm: multi-step — title → description → stake → participants → deadline → review. Inline validation.
  • VotePanel: vote buttons (valid/invalid) disabled if already voted. Show current tally. Optional comment.
  • ProofForm: URL input + file upload (preview). Drag/drop. Store in S3 or local /uploads.
  • Toast: global notification (success/error).
  • Header: user avatar, nav links, create wager button.

Real-time Strategy

Option A: SSE (simpler) — /api/v1/wagers/stream sends events on changes (proof added, vote cast, status update). Frontend EventSource updates store.

Option B: WebSocket — more flexible, needed for chat/comments. Use @bedframe/live or custom.

Initial: polling every 10s if SSE not available. Upgrade later.

Security Requirements

Authentication

  • bcrypt with cost 12
  • JWT access token (15min), refresh token (7d) stored httpOnly, sameSite=strict
  • CSRF protection (SvelteKit built-in)
  • Password reset flow (email)

Authorization

  • Only creator can edit/delete wager
  • Only participants can vote/submit proof
  • Admin role can resolve disputes (future)

Input Validation

  • Zod schemas server-side + client-side
  • Sanitize HTML in description (DOMPurify)
  • Max lengths: title(200), description(2000), stake(18 digits), participants(max 50)
  • Deadline must be future date

Rate Limiting

  • Create wager: 5/hour per user
  • Vote: 20/hour per user
  • API general: 100/min per IP

Headers

  • helmet (or SvelteKit built-in) — CSP, HSTS, X-Content-Type-Options, etc.

File Uploads

  • Validate mime type, size (<10MB)
  • Store outside static dir, serve via signed URL
  • Virus scan (optional)

Open Questions

  1. Should we use Dolt from day 1?
    Recommendation: start with SQLite for speed, plan migration to Dolt by Phase 4. Dolt introduces CLI ops complexity.
  2. Email notifications provider?
    Options: Resend, Postmark, AWS SES. Pick based on cost/volume. Phase 6.
  3. File storage for proofs?
    Options: local /uploads (simple), S3/R2 (scalable). If local, ensure backups. Phase 6.
  4. Realtime need?
    Initially polling OK (10s). If users want instant, add SSE. Avoid WebSocket unless chat feature confirmed.
  5. Admin UI?
    Initially CLI + Dolt CLI for admin tasks. If needed, add minimal admin panel later.
  6. Monorepo?
    No, keep single repo (SvelteKit handles API+frontend). Separate services unnecessary.
  7. Docker for dev?
    Optional. Use docker compose for DB only if Dolt. Local SQLite no container needed.
