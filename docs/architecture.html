System Architecture

Overview

SvelteKit monolithic app with embedded database. Monolith simplifies deployment while maintaining clean separation of concerns. Database can be SQLite (dev) or Dolt (prod) with minimal code changes.

Key Principles

  • Single deployable unit: SvelteKit builds to Node.js server; includes API routes, SSR, and static assets.
  • Server-side rendering: Initial page load is fast, SEO-friendly.
  • API routes co-located: src/routes/api/ serves JSON endpoints.
  • ORM abstraction: Prisma enables swap between SQLite and Dolt.
  • Type safety: TypeScript end-to-end; Prisma types auto-generated.
  • Edge ready: Can deploy to Vercel/Netlify edge functions if needed.

Component Interactions

1. User Registration / Login

  1. Client sends POST /api/v1/auth/register with email/password.
  2. Server validates input (Zod), checks email uniqueness.
  3. bcrypt hash password, create User record via Prisma.
  4. Generate session token (or JWT), set httpOnly cookie.
  5. Return user object (without password).

2. Create Wager

  1. Client navigates to /wagers/new (protected route).
  2. +layout.server.ts loads user session, redirects if not authenticated.
  3. Form validated client-side (Zod) and server-side (Zod).
  4. Server creates Wager + WagerParticipant rows in transaction.
  5. Broadcast SSE event wager:created to other connected clients.
  6. Redirect to wager detail page; populate store.

3. Submit Proof

  1. On wager detail page, participant clicks "submit proof".
  2. Modal/form opens; optional file upload (multipart/form-data).
  3. File stored to /uploads or S3; URL returned.
  4. POST /api/v1/wagers/:id/proofs with { link: url }.
  5. Server appends Proof record, updates wager status to SUBMITTED if first proof.
  6. SSE event proof:added broadcast.
  7. All clients update UI; proof appears in list.

4. Vote

  1. Each participant can vote once per wager.
  2. POST /api/v1/wagers/:id/votes with { isValid: boolean, comment? }.
  3. Server creates Vote; if all participants voted, auto-transition to RESOLVED (logic: majority valid? per rules).
  4. SSE event vote:cast broadcast.
  5. Clients update vote tally; disable vote buttons if already voted.

Database Schema Details

Normalization

  • WagerParticipant is join table (wager ↔ user). Enforces uniqueness.
  • Vote has unique constraint on (wagerId, voterId) to prevent double-vote.
  • Proof links to user and wager; multiple proofs allowed per user (history).
  • Comment also linked; cascade delete on wager deletion.

Indexes

  • Wager.status — filter by status (pending list)
  • Wager.deadline — deadline notifications, sorting
  • Wager.createdBy — user's wager list
  • Proof.wagerId — fetch proofs for wager
  • Vote.wagerId — tally votes
  • Comment.wagerId — discussion feed

Audit Trail Implementation

SQLite approach: Create trigger on each table (INSERT/UPDATE/DELETE) that writes to AuditLog table. AuditLog stores:

  • id (uuid)
  • tableName (string)
  • recordId (string)
  • action ('INSERT'|'UPDATE'|'DELETE')
  • oldValue (JSON, null for INSERT)
  • newValue (JSON, null for DELETE)
  • changedBy (userId, system for automated)
  • changedAt (timestamp)

Dolt approach: No need for custom triggers. Dolt internally tracks all changes. Use:

  • dolt diff 'HEAD' 'CARET' to see changes
  • dolt log --all for commit history
  • Export as CSV for reporting

Security Architecture

Authentication Flow (Session-based)

1. User submits credentials → /api/v1/auth/login
2. Verify password with bcrypt.compare
3. Create session record in DB (UserSession table) with:
   - id (uuid)
   - userId
   - tokenHash (random secret)
   - expiresAt
4. Set cookie: session_token=, httpOnly, sameSite=strict
5. On each request, +layout.server.ts:
   - Read cookie
   - Lookup session by tokenHash
   - Verify not expired
   - Attach user to locals
6. Logout → delete session record, clear cookie

Authorization Enforcement

  • Every API route calls requireAuth() helper; returns 401 if no session.
  • Resource checks: e.g., wager.createdBy === user.id for edit.
  • Admin role: future User.role column ('admin'|'user').

Technology Rationale

Why SvelteKit?

  • Less boilerplate than Next.js
  • Smaller bundle sizes (~30% smaller than React)
  • Compiler optimizations — no virtual DOM
  • Built-in API routes mean one framework
  • Great TypeScript support

Why Prisma?

  • Type-safe queries — compile errors if field wrong
  • Migrations included
  • Multi-db support (SQLite, Postgres, MySQL, SQL Server)
  • Introspection — generate schema from existing DB
  • CLI for seeding, migration, studio

Why Dolt?

  • Git operations on data: dolt branch, dolt merge, dolt diff
  • Time-travel queries: SELECT * FROM wagers FOR DATABASE_TAG 'main@2025-01-01'
  • Built-in data lineage
  • Can push/pull to remote DoltHub or self-hosted
  • SQL interface — no new query language

Why Tailwind?

  • Rapid UI development
  • No context switching to CSS files
  • Built-in dark mode variants
  • Consistent design tokens (spacing, colors)
  • JIT compiles only used classes → small bundle
