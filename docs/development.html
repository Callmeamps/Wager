Development Guide

How to set up, develop, test, and deploy the Wager Tracker project.

Prerequisites

  • Node.js 22+
  • npm or pnpm (npm recommended)
  • Git
  • Docker (optional, for Dolt)
  • Prisma CLI (installed via devDeps)
Port 4317: Dev server uses port 5173 by default. To avoid conflicts, configure PORT=4317 in .env.

Local Setup

1. Clone Repository

git clone https://github.com/your-org/wager.git
cd wager

2. Install Dependencies

npm ci

3. Environment Configuration

Create .env in project root:

# Database (SQLite file path)
DATABASE_URL="file:./data/wagers.db"

# JWT secret (generate: openssl rand -base64 32)
JWT_SECRET="your-secret-here-32-bytes-minimum"

# Server port (avoid 3000/5173)
PORT=4317

# Node environment
NODE_ENV="development"

# Allowed CORS origins (comma-separated)
ALLOWED_ORIGINS="http://localhost:4317"

# Optional: email (Resend API key for notifications)
RESEND_API_KEY=""
Note: For Dolt, use: DATABASE_URL="postgresql://user:pass@localhost:5432/wagers" and start Dolt container first.

4. Initialize Database

npx prisma generate
npx prisma migrate dev --name init

This creates prisma/migrations/ and applies schema to data/wagers.db. To seed demo data:

npx prisma db seed

Note: Add seed script to package.json: "prisma": { "seed": "tsx prisma/seed.ts" }

5. Start Development Server

npm run dev

Open http://localhost:4317. Hot reload enabled.

Coding Standards

TypeScript

  • Strict mode: strict: true in tsconfig.json
  • No any types. Use unknown if truly uncertain.
  • Prefer interfaces over type aliases for objects.
  • Export types from src/lib/types.ts when shared.

Svelte Components

  • Use <script lang="ts">
  • let props with explicit types
  • Reactive statements: $: computed = ...
  • Avoid imperative DOM manipulation; use bindings
  • Extract complex logic to lib/utils/ or stores
<script lang="ts">
  interface Props {
    wager: Wager;
  }
  let { wager }: Props = $props();

  let showProofForm = $state(false);
  let proofLink = $state("");

  async function submitProof() {
    await api.post(`/wagers/${wager.id}/proofs`, { link: proofLink });
    showProofForm = false;
  }
</script>

<div class="wager-card">
  <h3>{wager.title}</h3>
  <button onclick={() => showProofForm = true}>Submit Proof</button>
</div>

Naming Conventions

  • Files: kebab-case (wager-card.svelte, create-wager.ts)
  • Components: PascalCase (WagerCard, ProofForm)
  • Functions/vars: camelCase (submitProof, wagerList)
  • Constants: UPPER_SNAKE (MAX_UPLOAD_SIZE)
  • Types/interfaces: PascalCase (Wager, CreateWagerInput)

API Client

Use centralized src/lib/utils/api.ts wrapper:

import { api } from '$lib/utils/api';

// GET
const wagers = await api.get('/wagers?status=pending');

// POST with body
await api.post('/wagers', wagerData);

// Error handling: api throws on non-2xx, catch in component
try {
  await api.post(...);
} catch (err) {
  toast.error(err.message);
}

Validation

  • Server-side: Zod schemas in src/lib/server/validation.ts
  • Client-side: same schemas shared via zod package
  • Never trust client; always validate on API route

Database Migrations

Making Schema Changes

  1. Edit prisma/schema.prisma
  2. Run: npx prisma migrate dev --name descriptive_name
  3. Prisma creates new migration file in prisma/migrations/
  4. Review generated SQL (in migrations/.../migration.sql)
  5. Commit both schema.prisma and migration file

Resetting DB (dev only)

npx prisma migrate reset   # drops and reapplies all migrations
npx prisma db seed         # reseeds

Production Migration

npx prisma migrate deploy   # applies pending migrations, no dev prompts

Always test migrations on staging copy first.

Deployment

Build

npm run build

Output in .svelte-kit/output. Build includes server and static assets.

Docker Build

docker build -t wager:latest .
docker run -p 4317:4317 --env-file .env wager:latest

Deploy to Railway

  1. Push code to GitHub
  2. Create new Railway service, connect repo
  3. Set build command: npm run build
  4. Set start command: npm start
  5. Add environment variables (from .env)
  6. Add PostgreSQL service (for Dolt) or use mounted volume for SQLite (not recommended)
  7. Deploy

Deploy to Vercel

Vercel edge runtime may not support SSE or file system DB. Use Node.js server:

  • Set vercel.json with "functions": { "api/**/*.ts": { "maxDuration": 10 } }
  • Use Vercel Postgres (or external Dolt)
  • Set output: 'standalone' in svelte.config.js
  • Deploy; Vercel builds and runs as container
Important: Do not use SQLite file on Vercel (ephemeral filesystem). Use external DB.

Contributing Guidelines

Pull Request Process

  1. Create feature branch from main
  2. Make changes, add tests
  3. Run npm run check && npm run lint && npm run test locally; fix errors
  4. Commit with clear message (see Conventional Commits)
  5. Push and open PR against main
  6. Ensure CI passes
  7. Address review comments
  8. Merge only after at least one approval

Commit Message Format

type(scope): subject

  • type: feat, fix, docs, style, refactor, test, chore, build, ci
  • scope: auth, wager, vote, db, api, etc.
  • subject: brief description, imperative mood (e.g., "Add vote validation")

Example: feat(wager): add proof submission with file upload

Code Review Checklist

  • Does this change have tests?
  • Does it follow existing patterns?
  • Are there performance implications?
  • Are security considerations addressed (input validation, auth checks)?
  • Is the PR description clear?
  • Are migrations included if schema changed?
  • Are environment variables documented?

Troubleshooting

Database connection error

Check DATABASE_URL is correct, DB file exists (SQLite) or Dolt container running. Run npx prisma studio to verify connection.

Port already in use

Change PORT in .env to 4317 or another free port. Find process: lsof -i :4317, kill if needed.

Prisma client mismatch

Regenerate: npx prisma generate. Ensure @prisma/client version matches prisma version in package.json.

SSE not reconnecting

Check browser console for CORS errors. Ensure ALLOWED_ORIGINS includes client origin. SSE requires persistent connection; some proxies may timeout.

File uploads not working

Verify uploads/ directory exists and is writable. For S3, check bucket policy and API key.
